## Feature normalisation (17:12)

We learned from the past that everything we did involved creating nice values in our network. For images, we kept rescaling pixel values to ensure they were between 0 and 1. There’s a significant difference in scale between these variables, suggesting that age could span a wide range. Similarly, the number of visits is a smaller number. To train faster, we could use an optimizer with adaptive properties. However, we can address this by simply rescaling our numbers. As long as the relative relationships between the numbers in each data point remain consistent, we can adjust their magnitude. For example, if we divide all ages by 10, the numbers change, but the relationship between age and other factors remains intact. This makes the data more normally distributed, which is generally beneficial. Normalization is a common practice. We aim to maintain similar scales of numbers both input and output to avoid issues like exploding or vanishing gradients, which can hinder learning in deep learning models. Scaling numbers is straightforward and done frequently. Before actual learning begins, we preprocess our data to make it more like a normal distribution, centered around zero with a standard deviation of one. This is achieved by subtracting the mean and dividing by the standard deviation. This process is so common that various tools, such as scikit-learn’s StandardScaler, automate it. When using Keras, we can implement this as a normalization layer. It’s crucial to train the standard scaler on the training data and apply it to test and validation data without incorporating any test data information.

## Dealing with text and categorical data (30:15)

Encoding, we will discuss later. This is an example of encoding. Taking something and assigning it a number, such as zero, one, two, three, and so on. We call this ordinal encoding. It’s simple, right? We have one column, and we replace each value accordingly. I haven’t expanded my data unnecessarily. However, we’re assuming an order where a higher number represents a higher value, and zero represents the lowest. This isn’t always accurate, especially from a medical perspective. Therefore, we’ve imposed an order, which might not be meaningful. Adding assumptions without validation can lead to overfitting. One-hot encoding is a better approach. Instead of using a single column, we create new columns for each possible value, indicating presence or absence. For instance, for blood type, we might have columns for A, B, AB, and O, and assign values based on the patient’s blood type. This method avoids imposing an order and allows the model to learn relationships between different blood types. However, this approach can result in a large number of columns, leading to potential overfitting if the dataset is small. 

For diagnosis codes, we need a different approach since patients can have multiple conditions simultaneously. We could use ordinal encoding, assigning numbers to combinations of diagnoses, but we wouldn’t require the sums to equal one. Alternatively, we could treat each condition separately, creating new columns for each diagnosis. The model would then learn any correlations between these columns. Generally, this method allows the model to learn more effectively, reducing the burden on us to manually define complex relationships. If we have domain knowledge suggesting certain conditions should be treated differently, we can incorporate that into our preprocessing steps. However, the model can often learn these relationships on its own, making our job easier.

## Embeddings (36:54)

We're hinting ahead a bit to how we will handle text processing in a couple of weeks after we start the project. Natural language processing involves embedding techniques, which are crucial. Essentially, we agreed that we can add columns for each category in our dataset. However, adding numerous columns poses a risk of overfitting due to sparse data, especially in previous cases where many entries are zeros. Assuming a better scenario where most entries are zeros with a few exceptions, we can consider expanding these columns with symptoms such as headache and fatigue. Given that there are likely hundreds or thousands of different symptoms, this could result in hundreds of thousands of columns, which are inefficient and tedious to work with, particularly when overfitting occurs.

The question then arises: can we improve this approach? Yes, we can use embeddings, essentially creating a new abstract vector space that contains the necessary information but is learned through training. Instead of manually setting these columns, the model can determine the best columns. This process involves generating a lookup table that maps categories to vectors, representing them in an abstract space. The model learns these mappings during training, allowing for more efficient and manageable data representation.

For instance, instead of using integers to represent blood types, we can use floating-point values to create vectors. These vectors can point in specific directions based on the relationships between categories. If blood type A and B are related, their vectors will point in similar directions. Conversely, unrelated categories will point in orthogonal directions. Once we have these vectors, we can visualize them in a space, which helps in understanding their relationships.

In summary, we aim to avoid manual encoding and instead allow the model to learn these encodings automatically. The choice of the number of columns depends on the amount of data available; more data allows for higher dimensionality without overfitting. This technique, known as entity embeddings, can be applied to both categories and words as we move towards text processing. The dimensionality of the output can be adjusted based on the amount of data and the complexity of the task. While choosing the optimal dimensionality is often a matter of trial and error, it is essential for effective model performance.
