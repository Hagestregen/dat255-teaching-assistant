## Training decoder models (3:06)

These large language models that we are accustomed to are essentially next-word predictors or next-token predictors. They do not formulate a structured plan for sentences; instead, they predict one token at a time and then apply the complete attention mechanism across the input and the next word, repeating this process. This method is computationally expensive, though it works well. 

Now, there are newer models emerging that do not perform this exact one-token-at-a-time prediction. However, these models are not yet widely prevalent. If you use ChatGPT currently, you are performing next-word prediction. The advantage of this approach is that it’s relatively easy to visualize how such models could be trained. You can take a sentence as training data, mask the end of it, and then provide the model with the next word, allowing it to predict and then compute the loss to optimize the parameters. This process can be repeated until the end of the text. Typically, the training data consists of a significant amount of text, enabling the model to run this loop for an extended period. 

This method is often referred to as masked attention or causal attention, where the model is allowed to look only backward. An older, popular model called BERT, for example, masks random words during text generation, whereas we prefer to mask the head, allowing the model to look only at what comes before and predict the next word. This approach is suitable for text generation as we currently do it. 

One notable aspect is that when training with this method, you don’t need to annotate your data because the training data inherently provides the targets. You simply mask the next word and reveal it, making the entire training process self-supervised. In essence, the next word becomes the target for prediction throughout the training process.

## Generating text from a decoder model (6:42)

Quite important; let’s mention it again. For classification models, the last layer is a dense layer with the number of units equal to the size of the vocabulary, which we call the number of tokens the model needs to learn. This can often be tens of thousands for models like GPT-2, which I showed you earlier, having around 40,000 words. Most state-of-the-art models typically have a vocabulary size of several hundred thousand. The simplest approach is to use the softmax activation, which outputs values between zero and one. If a token is very likely, its value will be close to one, while unlikely tokens will be close to zero. To generate text, you run this in an autoregressive loop, similar to predicting steps in time series data. You feed the prediction from the previous step as the next input, and continue this process. If you start with a short prompt, the generated text can be very long. This method works similarly to sampling from a distribution and selecting the most probable word. 

Here’s an example from the old version of the book, which shows a distribution of possible next words. If you input “catch that on the,” the model might predict “and” or “the.” You can run this again, using your own outputs as the next inputs, and continue the process. You can run this indefinitely, deciding when to stop based on adding stop tokens or end-of-sequence tokens. If you encounter these tokens, the prediction stops, and you have your final text. Alternatively, you can keep running it, though the output might not be very human-like.

In practice, the scores you get are influenced by the initial inputs and past predictions. These scores form a list of 40,000 elements, giving a score for each possible token. Typically, the scores for words that don’t make sense in context will drop quickly. However, the list remains long, containing all possible tokens.

If you always pick the most probable word, you quickly get stuck in a loop, generating the same short sequence repeatedly. This is because the model becomes deterministic, producing the same output each time. A better approach is to predict multiple steps ahead and select the most probable sentence. This involves evaluating the likelihood of different branches of text and choosing the one with the highest overall probability. This method, known as beam search, tends to produce more probable-looking outputs but may still resemble machine-generated text due to the randomness introduced by the model's probabilistic nature. Beam search, we tried it for a little while but concluded relatively quickly that it wasn't really the way to go. However, we can explore an entirely different approach—embracing randomness. Instead of finding the most probable output text, we can simply select randomly from the scores. If the probability of a particular token is seven percent, we might sample that seven percent of the time. The rest of the time, we could sample another token based on the distribution of scores. Essentially, we set up a random number generator and sample according to the scores, meaning each run of the language model, even with the same scores, will produce different text. This approach starts to resemble human text, which, after all, can appear somewhat random.

However, if we have a large vocabulary of around ten thousand words, randomly selecting tokens occasionally results in unlikely words, leading to nonsensical sentences. To address this, we can implement top-k sampling, where we sample among the top hundred tokens with the highest scores, or even the top five, excluding the less likely ones. This ensures that the sampled tokens make relatively good sense.

Another approach we have taken is adjusted softmax. Instead of directly using the softmax function, which outputs values between zero and one, we adjust it using a parameter called temperature. This allows us to control how much randomness is introduced during sampling. By setting the temperature appropriately, we can fine-tune the sampling process to better match human-like text generation.

## Large language models (LLMs) (27:00)

Today and also for a bit tomorrow, let's look into the vague term of a large language model. All of our language models are currently based on the transformer architecture, with various implementations. Generally, large language models today have one billion or more parameters, often stated in billions (American numbering). Typically, the state-of-the-art models have hundreds of billions of parameters, which are considered quite heavy. A model around seven to eight billion parameters is often seen as a lightweight model, though these numbers are still absurdly large. Thus, it takes a significant amount of memory and RAM to store and compute these parameters reasonably.

## Training LLMs (28:12)

Let's look at the training business. In our notebooks, we train a small model since that's what we can afford. However, for larger models, they are typically divided into training steps. There are usually three rounds. The first round involves self-supervised learning, where we don't annotate our data ourselves but instead use a large chunk of text we've collected and mask the end sentence while revealing it. This results in a foundation model, which can be used for various purposes, though it's often not very useful on its own. Training a massive model on the entire internet can be expensive, costing millions of dollars on a data center. This isn't feasible for us, as it's too costly. Instead, we focus on tuning the model to specific tasks, such as answering questions. To achieve this, we need a dataset containing questions and answers or simulated conversations. The cost depends on the size of the domain we want to adapt to and the resources required.

Chatbots trained in this manner might initially provide responses by expanding on the input question, rather than giving direct answers. For example, if asked "How is the weather today?" it might continue the question, like "Is it cold or is it hot?" This behavior is due to the model being trained to predict the next word. To improve the chatbot's performance, we fine-tune the model specifically for the task. Fine-tuning can be a relatively inexpensive process, especially compared to training a massive model from scratch.

For instance, platforms like Hugging Face offer open-source models and fine-tuned versions. Continuous fine-tuning, where the model is further adjusted to fit specific cases, is particularly relevant for our needs. We can merge different models and perform quantization, which we plan to discuss further.

## Training data (33:21)

In cases where you need to tune a large language model to your specific task, one approach involves using training data. This training data typically comes from scraping the internet, focusing on high-quality sources such as Wikipedia and academic papers available on platforms like arXiv. For coding tasks, resources like the Archive are useful, containing academic texts. Additionally, sites like Stack Overflow provide valuable question-answer pairs.

For fine-tuning, you need annotated data with specific questions and answers. Large AI companies often keep their training data confidential, so obtaining similar datasets as OpenAI uses for creating ChatGPT is challenging. Making a good dataset for fine-tuning is difficult and something people tend to share more openly.

Reinforcement learning from human feedback is another method, where humans interact with the chatbot, providing ratings for responses. This process is costly because it requires human input to ask questions, provide answers, and rate the responses. To create a competitive chatbot like ChatGPT, significant human involvement is necessary.

## Distributed training (36:33)

Training a large language model requires accounting for the large amount of data needed, which necessitates using a data center with multiple compute units spread out. A popular open-source model, LLaMA 3.1, from a year and a half ago required 3.3 terabytes of RAM per GPU instance, making it impossible to fit on a single GPU. This means all training must occur in a data center.

We can split the data into batches, which can be processed separately on different computers. Running a small batch doesn't require much memory, and results can be merged at the end. Alternatively, different layers can be placed on different GPUs for separate training, allowing for different weights to be assigned to different parts of the model. Inputs can also be distributed across different computers, and during each training step, all learned information needs to be aggregated to ensure consistency across all nodes.

This distributed training approach requires significant compute power and substantial memory on each GPU, along with a high-speed network between nodes to aggregate weights. While this is beyond our current scope, TensorFlow offers methods for distributed training. For future machine learning engineers at big companies, frameworks like Hugging Face provide advanced techniques for distributed training.

For now, in this course, understanding the necessity of distributed training is key. Models exceeding a few gigabytes in size typically require this approach.
