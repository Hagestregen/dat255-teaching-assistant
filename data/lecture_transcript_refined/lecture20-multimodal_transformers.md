## LLM fine-tuning with limited resources (10:03)

Fine-tuning usually requires access to a massive data center and unlimited resources, which is not feasible for most projects. However, there are ways to achieve similar results with limited resources. One approach is to use the full pre-trained model and retrain it completely. This method is expensive due to the need for significant memory, including for the optimizer. A related issue is catastrophic forgetting, where fine-tuning can cause the model to forget previously learned information. To avoid this, we often opt for more targeted fine-tuning methods.

One technique is prompt tuning, where a smaller model is added in front of the large language model. This allows customization of the embedding space, making the model more context-specific without needing to retrain the entire model. Another technique is low-rank adaptation (LORA), which involves creating a lightweight parallel network that can be trained alongside the pre-trained model. This approach helps retain the original model’s knowledge while adding specific context, thus avoiding catastrophic forgetting.

Both methods reduce computational costs compared to full fine-tuning while maintaining the benefits of fine-tuning. These techniques can be applied to various scenarios, such as creating a chatbot for customer service or automatically documenting code bases. By using these approaches, developers can fine-tune models to meet specific needs without losing the foundational knowledge they gained during pre-training.

## Quantization (20:06)

We still have limited resources, as I mentioned. Memory is typically the limiting factor due to the large number of parameters that need to be stored in memory. For a single number, it usually requires 32 bits or 64 bits, depending on the specific task. In machine learning, we can reduce the number of bits needed for one number, allowing us to adjust the numerical precision. This means we can use fewer bits for weights and activations in each layer. For instance, instead of using 32-bit floats, we can use 16-bit floats, which allows us to store twice as many numbers. Reducing the number of bits doesn’t require significant modifications; however, for specialized systems like drones, we might use integers, such as 8-bit integers, to save space. These techniques help optimize numerical precision without significantly impacting performance.

For models like Hugging Face, the model names often include suffixes indicating quantization levels, such as "q4" for 4-bit quantization. When downloading a model, you might see options like "quantized to 8 bits," which provides better results but reduces the model size by a quarter. For smaller devices, such as phones, you might need to use quantized and optimized versions of the model, which may offer better compression ratios but at the cost of slightly reduced performance. 

Quantization is a trade-off between performance and resource usage. Depending on the application, this optimization can be crucial for running large language models locally on devices with limited resources.

## Knowledge distillation (25:54)

Knowledge distillation is essentially taking a smaller model and trying to make it mimic a larger model. You don’t need to go through quantization; you start directly with a smaller model. The idea is to set up a dialogue where the larger model generates answers, and the smaller model learns to replicate those answers. Essentially, the larger model serves as the teacher, and the smaller model learns to mimic its responses. The larger model is trained on all the training data, while the smaller model is trained specifically to replicate the larger model’s behavior. This process can work for tasks like classification and reasoning.

For more details, there are some papers available. Essentially, instead of humans providing instructions and answers, the system automates the entire process using a large language model to generate answers, and a smaller model learns to replicate these answers. While it might sound a bit strange, it does work. For example, on platforms like Hugging Face, you can find many examples of such models. Some companies, like Claude (a subsidiary of Anthropic), have been sued for distilling knowledge from their models into open-source models, effectively stealing knowledge from one model to another.

The training involves sending questions to the large model, receiving answers, and then using those answers to train the smaller model. This approach allows the smaller model to learn the essence of the larger model without needing to store all the parameters. Knowledge distillation is a powerful technique that can be applied to various tasks, including coding. I’ve tried the minimax model, which is good for coding tasks. When asked simple questions like “Who are you,” it responds with “I am Claude,” indicating some influence from the training data. ## Inductive biases (71:36)

We can say that we have an inductive bias when setting up my model. This is the assumption I make when building it. Some of these biases are quite obvious. For instance, if I use no deep learning at all, I might plot house prices against square meters, expecting a linear relationship. That could be a good idea, though it might not always be accurate. My assumption here is that the relationship will be linear. This leads to a simple model with only two parameters. The model is simple, but the assumptions are strong. If the actual relationship is curved, my model won’t perform well, having poor predictive power. However, the simplicity of the model means it requires less training data, simplifying the process in a way. This inductive bias applies to various models, including convolutional neural networks (CNNs). With CNNs, I assume that images are composed of pixels or neighboring pixels forming recognizable patterns. I also assume that moving parts around in an image should still represent the same object. These assumptions lead to translational equivalence, meaning the image’s structure remains consistent regardless of its position. Similarly, for recurrent neural networks (RNNs) used for time series, I assume that each neighboring time point has a sequential order, with the most recent time point being closer to the end. This assumption holds true when we understand the temporal sequence, but it’s still a form of bias. Transformers take this a step further by assuming a relational structure among data points. They don’t require the data to be ordered or follow a specific shape, making them highly adaptable. This generality allows transformers to be versatile across different tasks, which is why they are often chosen for foundational models capable of handling multiple tasks. However, this flexibility comes with a high demand for training data. To train a large language model, we need essentially the entire internet, and for a large vision transformer, we need millions of images. Thus, we must balance generality with specificity. If I’m tackling a highly specific task, a transformer might not be necessary. But for tasks that require versatility, a transformer is likely the best choice. In summary, we must carefully consider the assumptions we make when designing deep learning models, recognizing that some assumptions are beneficial while others may limit our model’s effectiveness.
